# GemmaQuiz - マルチプレイクイズゲーム

## プロジェクト概要

Ollama (Gemma4) によるAI問題生成を搭載した、最大8人対戦のリアルタイムマルチプレイクイズゲーム。
Unity 6 (URP) + Photon Fusion 2 (Hostモード) で構築する。

---

## 要件定義

### 1. ゲーム仕様

#### プレイヤー
- 最大8人同時対戦
- ホスト1名 + クライアント最大7名
- ホストがルーム作成・ゲーム開始を管理

#### クイズ形式
- **4択問題**（選択肢A/B/C/D）
- 将来的に他の形式（自由回答、○×など）を追加できる設計にする
- 問題は全て一般的な日本人が解答可能な内容に限定

#### ジャンル
| ジャンルID | ジャンル名 | 含まれるサブジャンル |
|-----------|-----------|-------------------|
| anime_game | アニメ・ゲーム | アニメ、マンガ、ゲーム |
| sports | スポーツ | 野球、サッカー、格闘技 等 |
| entertainment | 芸能 | 映画、音楽、TV、CM 等 |
| lifestyle | 生活 | グルメ、生活、ホビー、ファッション 等 |
| society | 社会 | 地理、政治、経済 等 |
| humanities | 文系学問 | 歴史、美術、文学、ことわざ 等 |
| science | 理系学問 | 物理、化学、生物、数学 等 |
| non_genre | ノンジャンル | 上記全てを包括 |

#### ゲームフロー
1. **ロビー**: ルーム作成/参加、プレイヤー一覧表示
2. **ジャンル選択**: プレイヤー指定またはランダムでジャンルを決定
3. **問題生成（ローディング）**: ホストがOllamaで選択ジャンルの問題10問を生成
4. **クイズ出題**: 1問ずつ全プレイヤーに同時出題（制限時間10秒/問）
5. **結果表示**: 各問の正解・各プレイヤーの回答・獲得ポイント表示
6. **次ジャンルまたは最終結果**: 選択された全ジャンル消化後、総合スコアで順位決定

#### スコアリング
- 正解時: 回答速度に応じて **10〜20点**
  - 1秒以内の回答: 20点（満点）
  - 制限時間（10秒）に近づくにつれ: 10点に漸減
  - 計算式: `score = 10 + 10 * max(0, (10 - elapsedSeconds) / 9)`（1秒以内は20点固定）
- 不正解: **0点**
- 未回答（タイムアウト）: **0点**
- 全ジャンル終了後の合計点で最終順位を決定

### 2. 技術仕様

#### 開発環境
- Unity 6 (URP 17.4.0)
- C# (.NET Standard 2.1)
- 対象プラットフォーム: PC (Windows) ※将来的にモバイル対応を視野

#### ネットワーク (Photon Fusion 2)
- Photon Fusion 2 を使用（Hostモード）
- ホストがサーバー兼プレイヤーとしてゲーム進行を管理
- NetworkRunner でセッション管理、INetworkRunnerCallbacks でイベント処理
- 問題データはホストからRPC ([Rpc(RpcSources.StateAuthority, RpcTargets.All)]) で全クライアントに配信
- 各プレイヤーの回答はRPCでホストに送信 → ホストが集計 → 結果を全員に配信
- セッション名 = ルーム名として使用

#### AI問題生成 (Ollama)
- ホストPCでOllamaをローカル実行（デフォルト: http://localhost:11434）
- モデル: **Gemma4**
- UnityWebRequestでOllama REST API (`/api/generate`) を呼び出し
- プロンプトでジャンル・問題数・4択形式を指定し、JSON形式で問題を取得
- 問題データ構造:
  ```json
  {
    "questions": [
      {
        "id": 1,
        "question": "問題文",
        "choices": ["選択肢A", "選択肢B", "選択肢C", "選択肢D"],
        "correct_index": 0,
        "explanation": "解説文"
      }
    ]
  }
  ```

#### UI方針
- 見栄えの良い演出を目指すが、テスト効率を優先
- まずはシンプルなUIで機能を完成させ、段階的に演出を追加
- URP対応のUI（uGUI / UI Toolkit検討）

### 3. アーキテクチャ方針

#### ディレクトリ構成（予定）
```
Assets/
├── Scripts/
│   ├── Network/          # Photon Fusion関連（接続、セッション管理、同期）
│   ├── Quiz/             # クイズロジック（問題管理、スコア計算、ゲーム進行）
│   ├── AI/               # Ollama連携（問題生成、プロンプト管理）
│   ├── UI/               # UI制御スクリプト
│   └── Data/             # データクラス（問題、プレイヤー情報など）
├── Prefabs/
├── Scenes/
│   ├── TitleScene.unity
│   ├── LobbyScene.unity
│   └── QuizScene.unity
├── UI/                   # UIアセット
└── Resources/
```

#### 設計原則
- クイズ形式を追加しやすいよう、問題表示・回答処理をインターフェースで抽象化
- ネットワーク層とゲームロジック層を分離
- Ollama通信は非同期処理（async/await または Coroutine）

### 4. 開発フェーズ

| フェーズ | 内容 | 優先度 |
|---------|------|-------|
| Phase 1 | Photon Fusion 2導入・セッション接続・ロビー基盤 | 最高 |
| Phase 2 | Ollama連携・問題生成パイプライン | 最高 |
| Phase 3 | クイズ出題・回答・スコア計算のコアループ | 最高 |
| Phase 4 | ジャンル選択・ゲーム進行全体フロー | 高 |
| Phase 5 | UI演出・エフェクト・ポリッシュ | 中 |
| Phase 6 | テスト・バグ修正・バランス調整 | 高 |

## 開発ルール

- スクリプトは `Assets/Scripts/` 以下に配置
- クラス名はPascalCase、変数名はcamelCase
- MonoBehaviourを継承するクラスはファイル名とクラス名を一致させる
- FusionのRPCメソッドには `[Rpc]` 属性を付与、NetworkBehaviourを継承するクラスで使用
- Ollama APIのエンドポイントは設定から変更可能にする
- Unity MCPツール経由でシーンやオブジェクトの操作が可能
